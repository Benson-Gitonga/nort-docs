Introduction NORT v1.0

NORT is a Polymarket AI assistant that automates prediction market analysis. It watches hundreds of active markets, surfaces the most interesting ones using a quantitative signals engine, and provides AI-generated trading advice through an agentic layer called OpenClaw. Everything runs in paper trading mode — no real funds, no real risk.

What is NORT?

Polymarket is a prediction market platform where people bet on real-world outcomes using real money. At any given moment there are hundreds of active markets — and browsing them manually to find an opportunity is impractical. NORT automates that entire process.

It fetches all active Polymarket markets, scores and ranks them by momentum and volume, exposes structured AI advice via OpenClaw, gates premium advice behind x402 micro-payments, and presents everything through a Next.js dashboard and a Telegram bot.

What NORT does in v1:
  • Fetches and caches all active Polymarket markets (auto-synced every 15 minutes)
  • Scores and ranks markets by momentum + volume spike via the Signals Engine
  • Provides free AI market analysis via the OpenClaw agent (OpenRouter)
  • Gates premium advice behind x402 micro-payment verification
  • Supports paper trades, wallet balances, P&L tracking, and a leaderboard
  • Exposes everything through a FastAPI backend, a Next.js dashboard, and a Telegram bot

Paper Trading Only

NORT v1 runs exclusively in paper trading mode. EXECUTION_MODE=PAPER is always on. No real funds are moved. The COMMIT_TX mode only writes receipts to Polygon testnet as an optional demo feature.

The Two Command Paths

Every request in NORT takes one of two paths. This is the most important architectural decision — it determines cost, latency, and whether the AI is involved at all.

Fast Path Telegram → FastAPI → SQLite → Response. Used for /trending, /signals, /market, /portfolio, /papertrade. Fast, cheap, no AI.
Agent Path Telegram → (x402 check) → FastAPI → OpenClaw → OpenRouter → LLM → Response. Used only for /advice and /premium_advice. Slower, costs tokens, produces structured JSON analysis.

The Rule Is Absolute

OpenClaw runs only for /advice and /premium_advice. It never runs for /trending, /signals, /market, /portfolio, or /papertrade. Breaking this rule adds cost and latency where it doesn't belong.

Installation

NORT runs fully locally via Docker Compose. One command starts the backend, populates the database, and makes all APIs available on localhost:8000. The dashboard is a separate Next.js app that runs on localhost:3000.

Prerequisites

  • Docker + Docker Compose — for the backend and database
  • Node.js 18+ — for the Next.js dashboard
  • Python 3.11+ — if running the backend outside Docker
  • OpenRouter API key — for the AI advice layer (OPENCLAW_TOKEN)
  • Privy App ID — for wallet auth in the dashboard

Environment Variables

Create a .env file in the project root. The backend reads these at startup.

# Backend
OPENCLAW_TOKEN=sk-or-...         # OpenRouter API key
EXECUTION_MODE=PAPER             # Always PAPER in v1
POLYMARKET_API_URL=https://gamma-api.polymarket.com

# Dashboard (apps/dashboard/.env.local)
NEXT_PUBLIC_API_URL=http://localhost:8000
NEXT_PUBLIC_PRIVY_APP_ID=clx...  # Privy app ID for wallet auth

Never commit secrets

The .env file is already in .gitignore. Never commit your OPENCLAW_TOKEN or PRIVY_APP_ID to the repository.

Running Locally

1. Start the backend
docker-compose up --build

This starts the FastAPI backend on port 8000, initialises the SQLite database, and syncs markets from Polymarket. The background sync loop will re-sync every 15 minutes automatically.

2. Start the dashboard
cd apps/dashboard
npm install
npm run dev

The dashboard is available at http://localhost:3000. It polls the FastAPI backend at NEXT_PUBLIC_API_URL.

3. Start the Telegram bot (optional)
cd apps/telegram-bot
pip install -r requirements.txt
python main.py

Verify it's running

Open http://localhost:8000 — you should see {"status":"online","message":"NORT Backend is active."}. The backend auto-docs are at http://localhost:8000/docs.

API Reference

All endpoints are served by the FastAPI backend on port 8000. Every route is registered both with and without an /api prefix for compatibility. Interactive docs are available at /docs.

Markets

GET /markets/ Returns all active markets. Query param: limit (default 100). Response includes markets[] array with id, question, category, current_odds, volume.
GET /markets/{id} Returns a single market by ID. Returns 404 if the market is not found or inactive.
GET /markets/refresh Triggers a manual sync from the Polymarket API. Returns count of markets synced. Called automatically on startup and every 15 minutes.

Signals

GET /signals/ Returns ranked market signals. Query param: top (default 20). Each signal includes market_id, score (0.0–1.0), reason (plain English), current_odds, volume.
Sample response:
[
  {
    "market_id": "market-001",
    "question": "Will BTC hit $100k by March?",
    "score": 0.7200,
    "reason": "Price moved +44.0% with 16.0x average volume.",
    "current_odds": 0.72,
    "volume": 89000,
    "category": "Crypto"
  }
]

Advice / Agent

POST /agent/advice Triggers OpenClaw for a specific market. Body: {"market_id": "...", "telegram_id": "...", "premium": false}. Returns structured JSON analysis. Returns 402 if premium: true and payment not verified.
POST /agent/x402/verify Verifies an x402 payment proof. Body: {"proof": "64-char-hex", "user_id": "..."}. Caches the verification for 24 hours.
Advice response schema:
{
  "market_id": "market-001",
  "summary": "...",
  "why_trending": "...",
  "risk_factors": ["...", "..."],
  "suggested_plan": "BUY YES",
  "confidence": 0.74,
  "disclaimer": "This is not financial advice. Paper trading only.",
  "tool_calls_used": ["openclaw_agent", "duckduckgo_search"]
}

suggested_plan is always one of: BUY YES, BUY NO, or WAIT. The backend enforces this — invalid LLM responses default to WAIT.

Paper Trading

POST /api/papertrade Logs a new paper trade. Body: {"telegram_user_id", "market_id", "market_question", "outcome": "YES"|"NO", "shares", "price_per_share", "direction": "BUY"}.
POST /api/trade/commit Optionally commits a trade receipt to Polygon testnet. Returns tx_hash. Only available if EXECUTION_MODE=COMMIT_TX.

Wallet & Leaderboard

GET /api/wallet/summary Returns the user's paper wallet. Query param: wallet_address. Returns paper_balance, net_pnl, total_trades, and a trades[] array.
GET /api/leaderboard Returns the ranked leaderboard. Query param: limit (default 50). Each entry includes rank, display_name, portfolio_value, net_pnl, win_rate, streak, xp, badge.
GET /api/leaderboard/me Returns the current user's rank entry. Query param: wallet_address. Returns null if user has no trades.
GET /api/user/stats Returns XP, level, streak, rank, and xpToNextLevel for a user. Used on the Profile and Achievements pages.
GET /api/user/achievements Returns all achievements with earned boolean and XP values. Achievements are computed from trade history.

Signals Engine

The signals engine is the intelligence layer between raw market data and useful rankings. It lives in services/backend/core/signals_engine.py and is called by the GET /signals endpoint. Without it, the system can only answer "here are all the markets." With it, the system answers "here are the markets you should actually pay attention to right now, and here's why."

Liquidity Filter

The first step is a hard gate. Any market with a volume below MIN_LIQUIDITY = 1000.0 is dropped before scoring. This prevents a market with two trades and $10 volume from scoring #1 due to a 100% price move — that's noise, not a signal.

MIN_LIQUIDITY = 1000.0

def passes_liquidity_filter(market: Dict) -> bool:
    volume = market.get("volume", 0)
    return volume >= MIN_LIQUIDITY

Scoring Pipeline

Each market that passes the liquidity filter is scored on two dimensions, then combined into a composite score.

Momentum Score (0.0 → 1.0)

Measures how much the market price has moved since the last snapshot. A large move signals that new information entered the market and informed traders are reacting.

def momentum_score(market: Dict) -> float:
    current  = market.get("current_odds", 0.5)
    previous = market.get("previous_odds", current)
    if previous == 0:
        return 0.0
    raw_change = abs(current - previous) / previous
    return min(raw_change, 1.0)  # capped at 1.0
Volume Spike Score (0.0 → 1.0)

Measures how unusual current volume is compared to the market's baseline average. A market with 3× its average volume confirms that many participants are reacting — not just one whale.

def volume_spike_score(market: Dict) -> float:
    current_volume = market.get("volume", 0)
    avg_volume     = market.get("avg_volume", 500)
    if avg_volume == 0:
        return 0.0
    ratio = current_volume / avg_volume
    return min(ratio / 5.0, 1.0)  # 5x avg = score 1.0
Composite Score

Both scores are weighted equally and combined. A market needs to show real activity across two dimensions to rank high, which reduces false positives.

MOMENTUM_WEIGHT = 0.5
VOLUME_WEIGHT   = 0.5

def composite_score(market: Dict) -> float:
    m = momentum_score(market)
    v = volume_spike_score(market)
    return round((MOMENTUM_WEIGHT * m) + (VOLUME_WEIGHT * v), 4)
Score ≥ 0.80 ◆ HOT — Strong momentum + unusual volume. High priority signal.
Score 0.50–0.79 ◆ WARM — Notable activity. Worth watching.
Score < 0.50 ◆ COOL — Low activity. Listed but deprioritised.

Reason Strings

Every ranked market must have a plain-English explanation of why it ranked. A score of 0.72 is meaningless to a human — "Price moved +44% with 16× average volume" is immediately understandable. This is what makes NORT feel intelligent rather than just mathematical.

def build_reason(market: Dict) -> str:
    current  = market.get("current_odds", 0.5)
    previous = market.get("previous_odds", current)
    volume   = market.get("volume", 0)
    avg_vol  = market.get("avg_volume", 500)

    if previous > 0:
        pct = ((current - previous) / previous) * 100
        price_str = f"Price moved {pct:+.1f}%"
    else:
        price_str = "Price data unavailable"

    if avg_vol > 0:
        ratio = volume / avg_vol
        vol_str = f"{ratio:.1f}x average volume"
    else:
        vol_str = "volume data unavailable"

    return f"{price_str} with {vol_str}."

Non-negotiable

Every signal must have a reason string. A score without an explanation is not acceptable — it's a hard rule in NORT's design principles.

OpenClaw Agent

OpenClaw is the AI advice layer. It's a structured reasoning agent — not a chatbot — that always produces the same output format regardless of which market it analyzes. It lives in services/agent/ and is called by the POST /agent/advice endpoint.

OpenRouter Integration

OpenClaw doesn't call OpenAI or Anthropic directly. It routes through OpenRouter, which acts as a unified gateway to any supported model. To swap models, change one line in agent.json — no code changes needed.

{
  "model": "anthropic/claude-3-haiku",
  "api_url": "https://openrouter.ai/api/v1/chat/completions"
}

Before calling the LLM, OpenClaw enriches the prompt with three data sources: the full market object (get_market(id)), the current signal snapshot (get_signals()), and recent DuckDuckGo news search results for the market's question.

Prompt Templates

The system prompt and user prompt are built in services/agent/prompt_templates.py. The system prompt enforces the output format — OpenClaw must always return valid JSON with exactly these keys:

{
  "summary":        "Plain English summary of the market situation",
  "why_trending":   "Explanation of the momentum and volume signals",
  "risk_factors":   ["Risk 1", "Risk 2", "Risk 3"],
  "suggested_plan": "BUY YES | BUY NO | WAIT",
  "confidence":     0.74
}

The backend parse layer (parse_response()) strips markdown fences, validates JSON, and enforces that suggested_plan is one of the three valid values. Invalid responses default to WAIT and never surface a bad value to the user.

x402 Payment Gate

Premium advice is gated behind x402 — an HTTP-native micro-payment protocol. The philosophy is pay-per-query rather than a monthly subscription.

/signalsFree. Anyone can see ranked markets.
/adviceFree. Basic AI analysis for all users.
/premium_advicePaid. Costs 0.10 USDC per query. Requires payment proof.

When a user sends /premium_advice, the bot checks whether payment has been verified for that user. If not, it replies with the payment prompt. The user gets an x402 proof string from the payment flow and submits it with /pay <proof>. The backend calls POST /agent/x402/verify, validates the 64-char hex proof, and caches the verification for 24 hours.

Demo mode

In v1, the x402 verifier accepts any valid 64-character hex string. Real on-chain verification via Base RPC is stubbed as # TODO in api/advice.py.

Data Models

All data is stored in a single SQLite database at data/assistant.db. Models are defined using SQLModel in services/backend/data/models.py. There are seven tables in total.

Core Tables

Market Populated by the market sync loop from Polymarket. Fields: id, question, category, current_odds, previous_odds, volume, avg_volume, is_active, expires_at.
AISignal Snapshot written after every /signals call. Fields: market_id, prediction, confidence_score, analysis_summary, timestamp.
User Created when a user first interacts. Fields: wallet_address (unique, indexed), telegram_id, username, created_at.
Trade Legacy trade table. Fields: user_id, market_id, outcome_selected, bet_amount, odds_at_time, status.
Payment x402 payment receipts. Fields: user_id, market_id, amount, tx_hash (unique), is_confirmed.

Paper Trading Tables

PaperTrade Table: paper_trades. Stores all paper trades. Fields: telegram_user_id, market_id, market_question, outcome (YES/NO), shares, price_per_share, total_cost, direction (BUY/SELL), status (OPEN/CLOSED/CANCELLED), tx_hash, pnl.
WalletConfig Table: wallet_config. Per-user paper wallet. Fields: telegram_user_id (unique), paper_balance (starts at $1,000), total_deposited.
LeaderboardSnapshot Table: leaderboard_snapshots. Daily stats per user. Fields: telegram_user_id, display_name, portfolio_value, net_pnl, total_trades, win_rate, snapshot_date.

Starting balance

Every new user starts with paper_balance = 1000.0 — a $1,000 paper USDC wallet. This is set in WalletConfig on first interaction and never changes unless a trade deducts from it.

Auth & Telegram

Authentication is handled by Privy (@privy-io/react-auth). Users can sign in with a wallet or Google. The useAuth hook abstracts the Privy client and exposes isAuthed, walletAddress, user, login, and logout.

The dashboard also supports running as a Telegram WebApp. The useTelegram hook reads window.Telegram.WebApp for user data and haptic feedback. When running in Telegram, the WebApp script is loaded via next/script in layout.jsx. The AuthSync component automatically syncs the Privy wallet address with the backend on first login.

Telegram Bot

The Telegram bot lives in apps/telegram-bot/. It is the primary user-facing interface for the fast path commands. It never reads from the database directly — all calls go through the FastAPI backend.

Command Reference

/trending Fast path. Returns the top markets by current volume from the SQLite cache. No AI. Instant response.
/market <id> Fast path. Returns full detail for a single market — odds, volume, category, expiry.
/signals Fast path. Returns the top 20 ranked signals with scores and reason strings. Calls GET /signals/?top=20.
/portfolio Fast path. Returns the user's paper wallet summary — balance, P&L, open trades.
/papertrade <id> <YES|NO> <amount> Fast path. Logs a paper trade. Example: /papertrade market-001 YES 50 bets $50 on YES.
/advice <id> Agent path. Calls OpenClaw for free AI analysis of the specified market.
/premium_advice <id> Agent path + x402. Returns premium AI analysis. Requires payment proof first.
/pay <proof> Submits an x402 payment proof. Calls POST /agent/x402/verify. If valid, unlocks /premium_advice for 24 hours.

Bot Architecture

main.py starts the bot and registers all command handlers from commands.py. The bot uses long-polling by default. Each command handler calls the appropriate FastAPI endpoint and formats the JSON response into a readable Telegram message.

Separation of concerns

The bot contains zero business logic. It only formats requests and responses. All scoring, trading, advice, and verification logic lives in the FastAPI backend. This means the dashboard and the bot always show identical data.

FAQs

Common questions about running, extending, and demoing NORT v1.

General

Is any real money involved?

No. NORT v1 is paper trading only. EXECUTION_MODE=PAPER is always on. No real Polymarket trades are placed. The optional COMMIT_TX mode only writes a testnet receipt to Polygon — still no real funds.

What does NORT stand for?

NORT is the product name — it's the brand, not an acronym. The app was named NORT in the codebase from the start.

How often does market data update?

Markets are synced from Polymarket on backend startup, then every 15 minutes via a background task (_market_sync_loop). You can also trigger a manual sync via GET /markets/refresh.

Can I run without a Telegram bot?

Yes. The dashboard is completely standalone. The Telegram bot is optional — the FastAPI backend and the Next.js dashboard are the core system.

Paper Trading

What is the starting balance?

Every new user gets $1,000 in paper USDC (paper_balance = 1000.0 in WalletConfig). This is set on first interaction and tracked against all trades.

How is P&L calculated?

P&L is stored on the PaperTrade record as pnl when a trade is closed. For open trades, P&L is estimated based on the current market odds vs the entry price. The dashboard reads this from GET /api/wallet/summary.

What is the "Commit" button in the dashboard?

The Commit button calls POST /api/trade/commit, which optionally submits a trade receipt to Polygon testnet and returns a tx_hash. This is a demo feature to show on-chain logging — no real money moves.

AI & Payments

How do I swap the AI model?

Change the "model" field in services/agent/agent.json to any model supported by OpenRouter (e.g. openai/gpt-4o, mistralai/mistral-7b). No code changes required.

Is the x402 payment real?

In v1, the x402 verifier accepts any valid 64-character hex string. Real on-chain verification (Base network RPC call) is stubbed and marked # TODO in services/backend/api/advice.py. For the demo, any 64-char hex works as a proof string.

Why does every advice response include a disclaimer?

It's a hard rule. Every advice response from OpenClaw must include: "This is not financial advice. Paper trading only." The backend enforces this — it's added by parse_response() and cannot be omitted by the LLM.

Can OpenClaw see my trade history?

Yes — for free advice, OpenClaw calls get_wallet_summary() as one of its context sources. This lets it tailor advice based on whether you already have an open position in the market.
